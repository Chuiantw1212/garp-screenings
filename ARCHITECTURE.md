# 架構決策

本文件概述了「篩選服務 (Screening Service)」的關鍵架構決策與資料流程。

## 資料獲取策略

為了提供一個反應快速、具備彈性且準確的金融數據服務（例如 PEGY 視圖），我們採用了三層式的資料獲取策略。當一個證券數據的請求進來時，系統會依照以下優先序來嘗試獲取資料：

### 第一層：Redis 快取 (Redis Cache)

-   **角色：** 系統的短期記憶。代表「我們最近剛看過的資料」。
-   **機制：** 在發出任何外部 API 請求之前，服務會先檢查 Redis 快取中是否存在對應證券代碼的、且尚未過期的資料。
-   **優點：**
    -   **速度：** 為頻繁請求的資料提供毫秒以下的反應時間。
    -   **成本與頻率限制：** 減少對外部 OpenBB API 的呼叫次數，節省成本並避免觸發 API 的頻率限制。

### 第二層：OpenBB API (即時資料)

-   **角色：** 與現實世界的對接。代表「當下發生的真實情況」。
-   **機制：** 如果在 Redis 快取中找不到資料（即「快取未命中 (Cache Miss)」），服務將會發出即時的 API 請求給 OpenBB，以獲取最新的資訊。
-   **優點：**
    -   **準確性：** 確保呈現給使用者的資料是新鮮的，能反映當前的市場狀況。
-   **後續動作：** 成功獲取資料後，結果會被存入 Redis 快取，以便快速回應後續的請求。

### 第三層：資料庫 (歷史存檔)

-   **角色：** 系統的最後一道防線。代表「我們過去所知道的最後真相」。
-   **機制：** 如果 OpenBB API 的呼叫失敗（由於網路問題、API 服務中斷或代碼無效），服務將會嘗試從我們自己的持久化資料庫（例如 PostgreSQL/MySQL）中，獲取該證券最近一次儲存的紀錄。
-   **優點：**
    -   **彈性與韌性：** 即使主要的外部資料來源不可用，服務依然能提供有意義的資料（儘管可能不是最新的）。這讓服務能夠優雅地降級，而不是直接回傳一個錯誤。
